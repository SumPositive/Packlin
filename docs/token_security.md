# アクセストークン延長・自動再取得に関するセキュリティ考察

## サーバー定義の有効期限を尊重する方針への回帰
- Keychainに保存したアクセストークンは、サーバーから通知された有効期限をそのまま利用するように戻しました。これにより、端末が第三者に取得された際でも、サーバーが設定した寿命を過ぎれば自動的に失効します。
- トークンの長期延命を行わないため、サーバー側の強制失効手段が無い環境でも漏えいリスクを最小限に抑えられます。
- 端末譲渡や下取り時には従来通りのアプリ削除・ログアウトを促す運用で十分となり、特別な注意喚起は不要になりました。

## エラー発生時に最新購入トランザクションを自動検証する影響
- `AiCreateView` が `missingAuthToken` や `tokenExpired` を検知すると、ユーザー操作無しで過去の購入トランザクションを再検証し新しいトークンを取り直すため、端末を第三者が操作できる状態であれば購入者の権限を継続的に再利用される可能性があります。
- 再検証処理は端末内の `AZUKI_CREDIT_PURCHASE_OPTIONS` に含まれる商品IDを走査し `Transaction.latest` を送信するだけなので、もし購入履歴が改ざんされていた場合（脱獄・Root化など）には、不正なレシートをサーバーへ繰り返し送って検証を誘発する面が残ります。
- 自動再検証はサーバー側へのリクエスト頻度を増やし、レート制限やサーバー側の検証ロジックが不十分な場合にDoSやリプレイ攻撃の踏み台となる懸念があります。

## 緩和策の例
- トークンにデバイス固有情報を紐づけ、サーバー側でデバイス単位の失効を行えるようにする。
- トランザクション再検証はユーザー操作（明示的な復元ボタン）をトリガーとし、端末状態のチェック（Jailbreak検知など）と併用する。
- 自動再検証を行う場合でもリトライ回数・待機時間を制限し、サーバー側でのレートリミットを徹底する。
